一、动态内存：vmalloc 方案细节优化

你的方向完全正确，但有几个关键细节需要补充：
1.1 vmalloc 在 KPM 中不一定可用

KernelPatch 模块运行在一个受限环境中，vmalloc / vfree 不是 KPM SDK 导出的函数。你需要通过 kallsyms 动态解析：
static void *(*kfunc_vmalloc)(unsigned long size) = NULL;
static void  (*kfunc_vfree)(const void *addr) = NULL;

// init 中:
kfunc_vmalloc = (typeof(kfunc_vmalloc))kallsyms_lookup_name("vmalloc");
kfunc_vfree   = (typeof(kfunc_vfree))kallsyms_lookup_name("vfree");

if (!kfunc_vmalloc || !kfunc_vfree) {
    // fallback: 用 KPM SDK 自带的 kp_malloc
    g_events = (svc_event_t *)kp_malloc(MAX_EVENTS * sizeof(svc_event_t));
} else {
    g_events = (svc_event_t *)kfunc_vmalloc(MAX_EVENTS * sizeof(svc_event_t));
}
1.2 vmalloc vs kp_malloc 的选择策略

特性	vmalloc	kp_malloc
最大分配量	理论无上限 (受虚拟地址空间限制)	依赖 KPM 内部分配器，通常 ≤ 几 MB
物理连续性	不要求 ✅	不确定 (取决于 KPM 实现)
是否 SDK 导出	❌ 需要 kallsyms	✅ SDK 原生
在中断上下文可用	❌ 不能在中断 /softirq 中调用	❓ 取决于实现
建议: 优先尝试 kp_malloc，因为它是 SDK 原生支持的。只有当 kp_malloc 对大块分配失败时，再 fallback 到 vmalloc。
1.3 你漏掉的关键点：memset 初始化

g_events = kfunc_vmalloc(MAX_EVENTS * sizeof(svc_event_t));
if (!g_events) return -ENOMEM;
memset(g_events, 0, MAX_EVENTS * sizeof(svc_event_t));  // 必须清零！
vmalloc 返回的内存不保证被清零（和 vzalloc 不同）。不清零的话，drain 输出时可能读到脏数据，JSON 中出现乱码。如果 vzalloc 在 kallsyms 中可用，直接用 vzalloc 更干净。
1.4 exit 时的释放保护

void kpm_exit(void) {
    // 先 unhook 所有 syscall，确保没有 hook 回调还在访问 g_events
    unhook_all_syscalls();

    // 加一个内存屏障，等所有 CPU 上正在执行的 hook 回调退出
    // 理想情况下用 synchronize_rcu()，但 KPM 中可能没有
    // 简易方案：sleep 一小段时间
    // kfunc_msleep(100);  // 需要 kallsyms 解析 msleep

    if (g_events) {
        if (kfunc_vfree)
            kfunc_vfree(g_events);
        else
            kp_free(g_events);
        g_events = NULL;  // 防止 use-after-free
    }
}
核心问题：unhook 和 free 之间必须保证没有 CPU 还在执行 hook 回调。否则 hook 回调访问已释放的 g_events → 内核 panic。最安全的做法是加一个 g_active 标志位 + 原子检查：
static volatile int g_active = 0;

// hook 回调入口:
if (!g_active || !g_events) return;

// exit:
g_active = 0;          // 先标记停止
smp_mb();              // 内存屏障，确保所有 CPU 看到
unhook_all_syscalls();
// 等待所有正在执行的回调退出 (简易: 延时)
// 然后 free
二、核心通信链路：长连接 Shell + 字符设备

2.1 libsu 长连接方案的细节

方向正确，但要注意几个坑：
问题 1：stdout 粘包
长连接 Shell 中，多次执行 kpatch ctl0 svc_monitor drain 的输出会粘在一起。你需要一个分隔符协议：
// App 端: 每次命令后追加一个唯一标记
val marker = "===END_${System.nanoTime()}==="
shell.newJob().add("kpatch ... ctl0 svc_monitor drain; echo '$marker'").submit()

// 读取时以 marker 为分隔，确定一次命令的输出边界
问题 2：Shell 断连重连
class PersistentShell {
    private var shell: Shell? = null
    private val lock = ReentrantLock()

    fun ensureShell(): Shell {
        lock.withLock {
            if (shell == null || !shell!!.isAlive) {
                shell = Shell.Builder.create()
                    .setFlags(Shell.FLAG_REDIRECT_STDERR)
                    .setTimeout(10)  // 10秒超时
                    .build()
            }
            return shell!!
        }
    }
}
2.2 字符设备方案：你需要解决的关键问题

这是终极方案，但在 KPM 环境下实现有特殊挑战：
问题 1：register_chrdev 不在 KPM SDK 中
同样需要 kallsyms 解析一系列函数：
static int (*kfunc_register_chrdev)(unsigned int major, const char *name,
                                     const struct file_operations *fops) = NULL;
static void (*kfunc_unregister_chrdev)(unsigned int major, const char *name) = NULL;
static struct class *(*kfunc_class_create)(struct module *owner, const char *name) = NULL;
static struct device *(*kfunc_device_create)(struct class *cls, struct device *parent,
                                              dev_t devt, void *drvdata, const char *fmt, ...) = NULL;
需要解析 5-6 个内核函数，其中任何一个在 kallsyms 中找不到都会导致方案失败。
问题 2：file_operations 结构体的版本兼容性
struct file_operations 在不同内核版本中字段不同。kernel 5.10.43 的 fops 和 5.15 的可能有差异。你需要确保结构体布局与目标内核完全一致。
问题 3：更现实的替代方案 — procfs
相比字符设备，/proc 文件系统的注册更轻量：
// 只需要两个函数
static struct proc_dir_entry *(*kfunc_proc_create)(
    const char *name, umode_t mode,
    struct proc_dir_entry *parent,
    const struct proc_ops *proc_ops) = NULL;
static void (*kfunc_remove_proc_entry)(const char *name,
    struct proc_dir_entry *parent) = NULL;

// 注册 /proc/svc_monitor
kfunc_proc_create("svc_monitor", 0444, NULL, &svc_proc_ops);
proc_ops 结构比 file_operations 简单得多，兼容性更好。APP 端只需 su -c "cat /proc/svc_monitor"。
问题 4：阻塞读取的实现
你说的 "阻塞读取" 需要等待队列：
static DECLARE_WAIT_QUEUE_HEAD(svc_waitq);  // 需要确认 KPM 能否用这个宏

// read 回调:
static ssize_t svc_read(struct file *file, char __user *buf,
                         size_t count, loff_t *ppos) {
    // 等待有新事件
    wait_event_interruptible(svc_waitq, event_count > 0 || !g_active);

    if (!g_active) return 0;  // EOF

    // 从 ring buffer 取事件，copy_to_user
    // ...
}

// hook 回调中，写入事件后:
wake_up_interruptible(&svc_waitq);
DECLARE_WAIT_QUEUE_HEAD 和 wait_event_interruptible 是宏，不是函数，它们展开后依赖的内核符号需要逐个确认在 KPM 中是否可用。
我的建议优先级：
方案	难度	可靠性	推荐
libsu 长连接 Shell	⭐ 低	⭐⭐⭐ 高	第一步做这个
/proc/svc_monitor	⭐⭐ 中	⭐⭐⭐ 高	第二步
/dev/svc_monitor 字符设备	⭐⭐⭐ 高	⭐⭐ 中	看需求再说

三、调用栈回溯的细节优化

3.1 你的描述缺少的关键前提

FP 链回溯需要从用户空间栈读取数据，这依赖 __arch_copy_from_user（之前讨论过的）。没有这个函数就无法实现。
3.2 回溯深度与性能的平衡

#define MAX_UNWIND_DEPTH  8   // 不要超过 8 层

int unwind_user_stack(hook_fargs6_t *args, unsigned long *bt, int max) {
    struct pt_regs *regs;
    unsigned long fp, frame[2];
    int depth = 0;

    if (!kfunc_copy_from_user) return 0;

    // 获取 pt_regs (has_syscall_wrapper=1 时从 args->arg0 获取)
    regs = (struct pt_regs *)((hook_fargs0_t *)args)->args[0];

    // 第0层: PC
    if (is_user_addr(regs->pc))
        bt[depth++] = regs->pc;

    // 第1层: LR
    if (depth < max && is_user_addr(regs->regs[30]))
        bt[depth++] = regs->regs[30];

    // FP 链回溯
    fp = regs->regs[29] & PAC_MASK;
    while (depth < max && is_user_addr(fp) && (fp & 0xF) == 0) {
        if (kfunc_copy_from_user(frame, (void __user *)fp, 16) != 0)
            break;
        unsigned long ret = frame[1] & PAC_MASK;
        if (!is_user_addr(ret)) break;
        bt[depth++] = ret;
        unsigned long next_fp = frame[0] & PAC_MASK;
        if (next_fp <= fp) break;  // 防止向下回溯或死循环
        fp = next_fp;
    }
    return depth;
}
3.3 JSON 输出格式

{
  "nr": 220,
  "name": "clone",
  "backtrace": [
    "0x7a3c001234",
    "0x7a3c005678",
    "0x7a3b00abcd",
    "0x7a3a009876"
  ],

    "libexec.so+0xabcd",
    "app_process64+0x9876"
  ]
}
backtrace 是裸地址（内核端输出），backtrace_resolved 需要结合 maps 在 APP 端解析（或内核端用缓存的 maps 解析）。建议内核端只输出裸地址，APP 端负责符号化—— 这样内核侧开销最小。
3.4 PAC 掩码的正确值

你 src/v4.0 中用的 0x0000FFFFFFFFFFFF 在 Pixel 6 上可能不够准确。Pixel 6 运行的是 ARMv8.2，PAC 使用高位 bit。具体掩码取决于 TCR_EL1.TBI 配置。Android 12 默认：
// 更保守的 PAC strip：取低 48 位
#define PAC_MASK  0x0000FFFFFFFFFFFFul

// 如果目标内核开启了 Top-Byte Ignore (TBI)，高 8 位可能被 tag 使用
// 此时应该取低 56 位
#define PAC_MASK_TBI  0x00FFFFFFFFFFFFFFul
Pixel 6 / Android 12 上建议用 0x0000FFFFFFFFFFFF，这个值是安全的。

4.2 自动清理策略

数据库不加限制会无限增长。需要设置上限：
// 保留最近 50 万条，超出时删除最旧的
@Query("DELETE FROM svc_events WHERE id NOT IN (SELECT id FROM svc_events ORDER BY timestamp DESC LIMIT 500000)")
suspend fun trimOldEvents()

// 每次批量插入后检查
suspend fun insertAndTrim(events: List<SvcEvent>) {
    eventDao.insertBatch(events)
    val count = eventDao.getCount()
    if (count > 500_000) {
        eventDao.trimOldEvents()
    }
}
